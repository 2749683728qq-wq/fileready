# DECISIONS — 重要决策记录

> 记录产品和技术上的重要取舍。按时间倒序排列。

---

## 决策 #1：项目从零初始化

- **日期**：2025-07-07
- **决策**：当前仓库为空，使用 Next.js App Router + TypeScript + Tailwind CSS 从零搭建
- **原因**：无已有代码需要兼容，可以直接使用最新稳定技术栈
- **影响**：Phase 0 包含项目初始化（Git、Next.js 基础配置），Phase 1 开始编写组件
- **替代方案**：无（仓库为空，无替代）

## 决策 #2：品牌名称暂缓确定

- **日期**：2025-07-07
- **决策**：品牌名称待后续确定，代码中使用 "File Upload Assistant" 作为中性占位
- **原因**：避免在未进行品牌调研、域名检查、商标查询前锁定名称
- **影响**：所有代码中使用中性名称，后续统一替换成本低
- **替代方案**：先随意取一个名称 → 被否决，因为改名成本高且可能侵权

## 决策 #3：首版不引入数据库

- **日期**：2025-07-07
- **决策**：P0 阶段不使用任何数据库，所有处理在浏览器端完成
- **原因**：核心功能均为浏览器端文件处理，无需持久化存储；用户账号和云端保存属于 P2
- **影响**：架构简单、隐私风险低、无服务器存储成本；但无法提供历史记录功能（P1 考虑 IndexedDB 本地存储）
- **替代方案**：使用轻量数据库 → 暂不需要，P2 再评估

## 决策 #4：HEIC 支持推迟到 P1

- **日期**：2025-07-07
- **决策**：首版不包含 HEIC 转换功能
- **原因**：HEIC 浏览器兼容性不稳定，部分浏览器不支持解码。需要先验证依赖稳定性
- **影响**：P0 中遇到 HEIC 文件时显示"格式暂不支持"并提供明确说明
- **替代方案**：使用服务器端转换 → 被否决，违反本地处理优先原则

## 决策 #5：广告组件预留但不加载

- **日期**：2025-07-07
- **决策**：Phase 0-7 中预留 AdSense 代码架构，但默认关闭
- **原因**：AdSense 审核需要真实内容和稳定网站，不是开发阶段的前置条件
- **影响**：创建占位组件和配置开关，开关关闭时不发送任何请求
- **替代方案**：上线后再加广告代码 → 被否决，预留架构成本低且避免后期大改

## 决策 #6：P1 的"粘贴要求"功能使用规则解析而非 AI

- **日期**：2025-07-07
- **决策**：解析用户粘贴的上传要求时，第一版使用可解释的规则解析
- **原因**：不依赖付费 AI 接口；规则解析可解释、可调试、成本为零
- **影响**：解析能力受限于预设规则，复杂/非标准表述可能无法识别；需用户确认解析结果
- **替代方案**：使用 GPT API → 被否决，违反"不依赖付费 API"原则且增加隐私风险

## 决策 #7：使用 pdf-lib 处理 PDF

- **日期**：2025-07-07
- **决策**：浏览器端 PDF 处理首选 `pdf-lib`
- **原因**：纯 JavaScript 实现、无外部依赖、支持合并/拆分/旋转/页面操作、包体积可控（~300KB gzipped）、维护活跃
- **影响**：PDF 渲染/预览可能需要额外方案（pdfjs-dist 用于预览）
- **替代方案**：pdfjs-dist 全套 → 包体积过大；服务器端处理 → 违反本地处理原则
